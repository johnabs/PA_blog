The JuliaLang Paradox

Posted Jan 29, 2020

The beginning of this post is going to sound ridiculous, but I promise it is 100% true. One night went to bed trying to remember the name of an interesting programming language that I heard about a while back. It supposedly had a lot of super nifty features, and was some strange hybrid of multiple languages that still somehow worked. I was intrigued, but couldn’t remember, so I decided to just go to sleep and figure it out in the morning. Well I had a strange fever dream, woke up at 5 A.M remembering the name of the language, Julia, and then I immediately went to my desk, downloaded the packages, and was testing it out. Let me tell you that was the best possible thing that could have happened to me that day in terms of my programming career and love of data science.

Julia is a scrappy new language that made its debut from the folks over at MIT, and it does almost everything right. It’s differentiable, it does pretty printing of data frames, it has multiple subsetting methods, it’s got amazing syntax, it has Hadamard operations FOR ANY FUNCTION YOU WRITE, and people are writing new packages for it by the day. There’s a research group dedicated towards using it for differentiable machine learning, which really speeds up the training process, there’s a professor who wrote a book in it, and despite the simple syntax, and ease of integration with other languages, it’s blazing fast. I would love to keep hyping it up, but I think that would be a bit much. Just kidding, it deserves it! It integrates beautifully with Jupyter notebooks for a more in-between editing mode for testing individual chunks at a time without needing to use the REPL or a script, it has a paper/book writing environment similar to RMarkdown (which is also a weak-point I’ll discuss later), can run C and Fortran code natively, and can call Python, R, and all sorts of other languages with some wrappers, then read those outputs into Julia native types and benefit from the speed of Julia and the flexibility of the near-infinite python and R libraries out there. Finally, you can develop GPU code entirely in Julia, which drastically improves its use for machine learning as well.

I CANNOT get enough of this amazing language which is this weird, beautiful hybrid of Python, Matlab, R, C and more. And the reason I named this article “The JuliaLang Paradox”, is because it almost seems to fly in the face of the idea that “too many chefs spoil the broth”. It flawlessly strips the grain from the chaff of other languages, and integrates the grains together beautifully into this sort of programming seed/nut loaf that just happens to go great with some metaphorical butter (or maybe some syntactic sugar…I’ll see myself out). However, despite it’s beauty, ease of use, and flexibility, it does have some downsides, at least for my applications.

Primarily the issue I was referring to above is known as the “time to first plot” issue. The way Julia is compiled, it only pre-compiles packages when you’re going to use them, which, in theory, should save on time in the long term. However, this increases run-time of scripts that only do a small thing like generate a single plot from data. I collected a bit of data here and found that to make an identical document in RMarkdown vs Weave from the same CSV file, using the same plotting package. R took 1.53 seconds, whereas weave took 67.59 seconds. That difference on a small project like this is HUGE, and I clearly wouldn’t recommend using it for single dummy documents like this. However, this time difference improves as the documents eventually get longer, as the real kicker for Julia is the time to the first plot, whereas the rest of the plots move much quicker in comparison. So eventually, especially for thesis documents or long research papers, Weave should hypothetically come out on top. (I may test this in the future, but I really don’t want to migrate my whole thesis into Weave at the moment).Additionally, in Julia 1.4, when it gets released, this issue should be addressed and allow for much faster compile times! So not much longer until we get some comparable performance on the short side of things too!

Finally, I’ll let you know that I’ve been really flexing my Julia muscles trying to write some interesting software. On my Github, I have a wrapper for the Ally-Invest API that you can use to try your hand at algorithmic trading on Ally if you’re interested. No, I’m not even remotely sponsored by them, in-fact they decided to ignore my emails about some API issues they have, but I definitely found their API easier to use than Robinhood’s which had 0 official documentation. I’ll leave links below for Julia, it’s IDE (or vim plugins if you took my article on CLI-apps to heart), as well as a link to that API wrapper for those of you who may be able to use it. It’s still in progress, but if you like it and want to donate or something, I’ll add my ETH Wallet there, and I have a brave certified creator’s account if you want to do so that way (seamless shilling, I know).
